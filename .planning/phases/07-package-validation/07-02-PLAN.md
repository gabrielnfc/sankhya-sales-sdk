  <action>
    Fix all 10 `exactOptionalPropertyTypes` errors. The fix pattern is: when assigning a value that could be `undefined` to an optional property, either add `| undefined` to the target type or use a nullish coalescing fallback. Here are the exact fixes:

    **1. src/core/auth.ts line 33** — `this.cacheProvider = cacheProvider;`
    The field `private readonly cacheProvider?: TokenCacheProvider` is optional. The parameter `cacheProvider?: TokenCacheProvider` passes `| undefined`. Fix: change field declaration to `private readonly cacheProvider: TokenCacheProvider | undefined;` (line ~15).

    **2. src/core/errors.ts line 16** — `this.statusCode = statusCode;`
    The field `readonly statusCode?: number` receives `statusCode?: number` parameter. Fix: change field declaration at line 9 from `readonly statusCode?: number;` to `readonly statusCode: number | undefined;`.

    **3. src/core/errors.ts lines 90-91** — `this.tsErrorCode = tsErrorCode; this.tsErrorLevel = tsErrorLevel;`
    Fields `readonly tsErrorCode?: string` and `readonly tsErrorLevel?: string` receive optional params. Fix: change declarations at lines 77-78 from `readonly tsErrorCode?: string;` and `readonly tsErrorLevel?: string;` to `readonly tsErrorCode: string | undefined;` and `readonly tsErrorLevel: string | undefined;`.

    **4. src/core/gateway-serializer.ts line 168** — `totalRecords: Number.parseInt(entities.total ?? '0', 10) || undefined`
    The `|| undefined` produces `number | undefined` but `PaginatedResult.totalRecords` is `totalRecords?: number`. Fix: change the expression to use a fallback: `totalRecords: Number.parseInt(entities.total ?? '0', 10) || 0`. If 0 is semantically wrong (meaning "unknown total"), instead update `src/types/common.ts` line 20 from `totalRecords?: number;` to `totalRecords?: number | undefined;`. Use the `| undefined` approach since the field genuinely can be unknown.

    **5. src/core/http.ts line 122-127** — The `fetch()` call passes `body: string | undefined` and `signal: AbortSignal | undefined` but `RequestInit` properties are optional (not `| undefined`). Fix: spread conditionally instead of passing undefined:
    ```typescript
    const response = await fetch(url, {
      method,
      headers,
      ...(body !== undefined ? { body: JSON.stringify(body) } : {}),
      ...(combinedSignal ? { signal: combinedSignal } : {}),
    });
    ```

    **6. src/core/pagination.ts line 14** — `totalRecords: data.length` assigned to `totalRecords?: number`. After fixing common.ts to `totalRecords?: number | undefined`, this is fine as-is.

    **7. src/core/pagination.ts line 21** — `totalRecords: Number.parseInt(pagination.total, 10) || undefined` same pattern as #4. After fixing common.ts, this is fine.

    **8. src/core/pagination.ts line 43** — same pattern as #4 and #7. After fixing common.ts, this is fine.

    **9. src/core/pagination.ts line 66** — `return { data, pagination }` where pagination is `RestPagination | undefined` but return type has `pagination?: RestPagination`. Fix: update the return type annotation of `extractRestData` to use `pagination?: RestPagination | undefined` OR change the return to spread: `return { data, ...(pagination ? { pagination } : {}) };`. Use the explicit undefined approach for consistency — change the return type at line 51 to `{ data: T[]; pagination?: RestPagination | undefined }`.

    **10. src/resources/cadastros.ts line 327** — `codigoNatureza: row.CODNAT ? Number(row.CODNAT) : undefined` assigned to `ModeloNota.codigoNatureza?: number`. Fix: update `src/types/cadastros.ts` line 124 from `codigoNatureza?: number;` to `codigoNatureza?: number | undefined;`. Same for line 126: `codigoCentroResultado?: number;` to `codigoCentroResultado?: number | undefined;`.
